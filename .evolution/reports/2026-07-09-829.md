## Problem

Issue #829 asked to fix a deadlock risk in the memory tool's file locking. The existing `_file_lock` context manager used `fcntl.flock(fd, LOCK_EX)` — a blocking call with no timeout. If another process held the lock indefinitely (e.g., a crashed cron job or a hung subagent), the memory tool would block forever, stalling the entire agent conversation.

## Root Cause

`_file_lock` in `tools/memory_tool.py` used blocking `fcntl.flock(LOCK_EX)` with no timeout mechanism. The lock is acquired at 5 call sites (lines 588, 676, 746, 824, 1013) — all of which could deadlock under contention.

## Solution

Added a new non-blocking lock acquisition method `_acquire_fcntl_lock` that uses `LOCK_EX | LOCK_NB` (non-blocking) with a retry loop:

- Attempts acquisition every 0.1s up to a 10s timeout (100 retries)
- Raises `TimeoutError` if the lock can't be acquired within the timeout
- The existing `_file_lock` context manager delegates to this method, so all 5 call sites automatically get bounded acquisition
